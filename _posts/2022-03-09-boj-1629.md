## **📝 문제**

*[B.O.J. #1629](https://www.acmicpc.net/problem/1629)*

### **문제**
---

 자연수 A를 B번 곱한 수를 알고 싶다. 단 구하려는 수가 매우 커질 수 있으므로 이를 C로 나눈 나머지를 구하는 프로그램을 작성하시오.
<br>
<br>

### **입력**
---

 첫째 줄에 A, B, C가 빈 칸을 사이에 두고 순서대로 주어진다. A, B, C는 모두 2,147,483,647 이하의 자연수이다.
<br>
<br>

### **출력**
---

 첫째 줄에 A를 B번 곱한 수를 C로 나눈 나머지를 출력한다.
<br>
<br>

## **📌 주어진 조건 & 구하고자 하는 답**

 문제에서 주어진 조건 중 ***2,147,483,647***이라는 숫자는 자료형 int의 upper bound이다. 구하고자 하는 답은 ***A를 B번 곱한 수를 C로 나눈 나머지***이다.

## **💭 사고의 흐름**

 문제에서 구하고자 하는 답은 주어진 자연수 A, B, C에 대해서 A를 B번 곱한 수를 C로 나눈 나머지이다. 그리고 이 각각의 자연수들의 범위는 자료형 int가 표현할 수 있는 모든 자연수이다. 단순하게 생각하면 말그대로 A의 B제곱을 구해서 C로 나누어 나머지를 구하는 코드를 작성할 수 있겠지만, 이는 위험한 생각이다. 문제에서 주어지는 A, B, C의 범위는 2,147,483,647이하의 자연수이다. 만약 A나 B나 C가 2,147,483,647 또는 매우 큰 수로 주어진다면 정수 자료형이 표현할 수 있는 범위를 훨씬 넘어서게 되어 답을 구하는 과정에서 오류가 생기게 된다. 이러한 제한사항을 어떻게 극복할 수 있나를 생각해보면 다음과 같은 Modular Arithmetic을 떠올릴 수 있다.<br>
```
a × b mod m = ( ( a mod m ) × ( b mod m ) ) mod m
```
 위 공식을 이용하면 A의 B제곱을 적절히 분할하여 답을 구할 수 있다. 즉, A의 B제곱이라는 큰 수를 계속해서 작게 나누어 A까지 도달한 뒤, A를 C로 나눈 나머지를 이용하여 곱셈 및 modular 연산을 통해 답을 구할 수 있는 것이다. 이러한 과정을 통해서 문제를 해결하면 정수 자료형이 허용하는 범위 내에서 연산이 가능하다. 이러한 문제 해결 패러다임을 ***Divide and Conquer***, 즉, ***분할 정복***이라고 한다. 분할 정복은 대게 ***재귀***라는 또 다른 패러다임을 통해 구현한다. 재귀는 수학적 귀납법에 기초하기 때문에 이를 통해 구현해야 할 코드를 생각해볼 수 있다.<br>
```
A mod C를 구할 수 있다.
A^k mod C를 구할 수 있다면,
A^2k mod C와 A^(2k+1) mod C또한 구할 수 있다.
```
 임의의 A, B, C가 주어졌을 때, 문제의 답을 func(A, B, C)라고 하자. 이 때, func(A, B / 2, C)를 알아낸다면 func(A, B, C) 또한 알아낼 수 있다. 따라서, 지수에 해당하는 부분을 계속해서 나누어 
Base Case에 도달하면 A mod C에 해당하는 값을 리턴하는 방식으로 코드를 구현하면 쉽게 답을 구할 수 있다.
```cpp
#include <iostream>

int func(int A, int B, int C);

using namespace std;

int main(void){
  ios::sync_with_stdio(0);
  cin.tie(0);
  
  int A, B, C = 0;

  cin >> A >> B >> C;

  cout << func(A, B, C) << "\n";
  
  return 0;
}

int func(int A, int B, int C){
  if(B == 1){

    return A % C;
  }

  long long val = func(A, B / 2, C);
  val = val * val % C;

  if(B % 2 == 0){
    
    return val;
  }
  val = val * A % C;

  return val;
}

```

## **🤔 되돌아보기......**

 이 문제를 포스트로 작성해야겠다는 생각을 한 이유는 재귀라는 패러다임을 대할 때 거쳐야하는 사고 과정이 이 문제에 잘 담겨있다고 생각했기 때문이다. 솔직히 처음 이 문제를 풀었을 때는 위에 작성된 사고 과정처럼 구현 원리를 깔끔하게 생각해내지 못했다. (심지어 overflow에 대해서, modular arithmetic에 대해서도 생각해내지 못함) 재귀를 이용할 때, 수학적 귀납법을 통해 구현 방법을 설계하는 것, 그리고 가장 중요한 Base Case 설정과 재귀 과정이 Base Case로 수렴하는지 확인하는 것, 이 두 가지를 꼭 명심해야겠다.
